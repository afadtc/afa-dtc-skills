# 反模式与边界处理

> 本文件为当前生效的模块参考资料，供执行时按需加载；用于扩展自然搜索增长中的反模式、边界条件与风险控制规则。若与当前模块 `SKILL.md` 或 `_system/` 全局协议冲突，以系统级规则为准。

---

## 一、绝对禁止的操作

```
黑帽/灰帽 SEO（零容忍）：
├── ❌ 购买链接（包括付费发布、链接交换计划）
├── ❌ 隐藏文本或链接（CSS 隐藏、白底白字）
├── ❌ 关键词堆砌（在内容、Meta 标签或 Alt 文本中不自然地重复关键词）
├── ❌ 门页（Doorway Pages）：为搜索引擎创建的无价值页面
├── ❌ 自动生成的垃圾内容（无人工审核的纯 AI 批量生成）
├── ❌ 链接农场和 PBN（私人博客网络）
├── ❌ 负面 SEO 攻击（向竞品发送垃圾链接）
├── ❌ 抄袭或刮取他人内容
└── ❌ 伪装（Cloaking）：向搜索引擎和用户展示不同内容
```

## 二、常见错误模式

```
内容错误：
├── ❌ 为关键词写作而非为用户写作
│   └── ✅ 正确：先满足用户意图，再自然融入关键词
├── ❌ 忽视搜索意图（用博客文章竞争交易型查询）
│   └── ✅ 正确：匹配搜索意图选择正确的页面类型
├── ❌ 内容薄弱（200 字的产品描述、100 字的分类页描述）
│   └── ✅ 正确：产品页 500-1,500 字，分类页底部 300-600 字
├── ❌ 关键词蚕食（多个页面竞争同一关键词）
│   └── ✅ 正确：一个关键词 = 一个目标页面，明确映射
└── ❌ 忽视内容刷新（发布后不再更新）
    └── ✅ 正确：每季度审查，流量下降 > 20% 立即刷新

技术错误：
├── ❌ 忽视 Core Web Vitals（"速度不重要"）
│   └── ✅ 正确：速度是排名因素，也直接影响转化率
├── ❌ 忽视移动端体验（"大部分用户用桌面端"）
│   └── ✅ 正确：Google 使用移动端优先索引
├── ❌ 结构化数据不完整（只有基础 Product Schema）
│   └── ✅ 正确：部署完整的 Product + FAQ + Breadcrumb + Organization
├── ❌ 忽视 robots.txt 和 sitemap
│   └── ✅ 正确：定期检查，确保爬取效率
└── ❌ 不处理重复内容（变体产品、分面导航）
    └── ✅ 正确：使用 canonical 标签和 robots.txt 精确处理

策略错误：
├── ❌ 只关注高搜索量的头部词
│   └── ✅ 正确：长尾词通常转化率更高，竞争更低
├── ❌ 忽视品牌 SEO（"品牌词自然就排第一"）
│   └── ✅ 正确：品牌词排名需要主动维护，竞品可能投放品牌词广告
├── ❌ SEO 与其他渠道割裂
│   └── ✅ 正确：SEO 与 PPC、社交媒体、邮件营销协同
├── ❌ 期望立竿见影的效果
│   └── ✅ 正确：SEO 是 6-12 个月的长期投资，但回报是复利型的
└── ❌ 追求虚荣指标（排名、流量）而非商业成果
    └── ✅ 正确：始终以收入、转化、ROI 为最终衡量标准
```

## 三、边界处理规则

```
当用户要求执行黑帽策略时：
├── 明确拒绝，解释风险（Google 惩罚、排名清零、品牌声誉损害）
├── 提供正当的替代方案
└── 示例："购买链接的短期效果不可持续，且有被 Google 惩罚的风险。我建议通过数字公关和可链接资产来自然获取高质量链接。"

当用户提供的数据不足时：
├── 明确说明需要哪些数据
├── 提供获取数据的方法（如何设置 GSC、如何导出数据）
├── 在数据有限的情况下，基于可用信息提供初步建议
└── 标注哪些建议需要更多数据来验证

当 SEO 建议与品牌调性冲突时：
├── SEO 不应牺牲品牌体验
├── 在品牌调性框架内优化（参考 brand-brain 的 voice-and-tone.md）
├── 示例：如果品牌调性是极简，不要为了 SEO 在页面上堆砌大量文字
└── 找到平衡点：底部购买指南可以在不影响首屏体验的情况下增加 SEO 内容

当 SEO 目标与短期商业目标冲突时：
├── 说明 SEO 是长期投资，短期可能需要 PPC 补充
├── 提供"快速见效"的 SEO 策略（如优化现有高流量页面）
├── 建议与 PPC 协同：PPC 覆盖短期，SEO 建设长期
└── 用 ROI 计算展示 SEO 的长期价值
```

## 四、SEO 专有降级策略

### Level 1（缺少 GSC/Analytics 数据）

```
SEO 降级 Level 1：
├── 触发条件：用户没有提供 GSC 或 Analytics 数据
├── 策略调整：
│   ├── 技术审计：基于页面源代码和公开工具（PageSpeed Insights, Rich Results Test）
│   ├── 关键词研究：基于 Google Autocomplete + PAA + Keyword Planner
│   ├── 竞品分析：基于 SERP 可见信息 + SimilarWeb 免费版
│   └── 所有建议标注「基于公开数据，如有 GSC 数据可进一步精确」
└── 不降低服务质量，只调整数据来源
```

### Level 2（缺少品牌定位和受众画像）

```
SEO 降级 Level 2：
├── 触发条件：用户没有品牌定位信息
├── 策略调整：
│   ├── 关键词策略：基于品类通用关键词，而非品牌差异化关键词
│   ├── 内容策略：以信息型内容为主，避免需要品牌调性的内容
│   ├── 语调：使用中性专业语调
│   └── 标注「建议完成品牌定位后，可进一步优化 SEO 内容策略」
└── 不降低服务质量，只调整内容方向
```

### Level 3（仅知道产品名称）

```
SEO 降级 Level 3：
├── 触发条件：用户只提供了产品名称
├── 策略调整：
│   ├── 提供品类通用 SEO 最佳实践
│   ├── 基于产品名称推断可能的关键词方向
│   ├── 给出通用的技术 SEO 清单
│   └── 标注「以上为通用建议，提供更多信息可获得定制方案」
└── 仍然提供有价值的输出
```

## 五、供应链模式适配

```
当 supply_chain_mode = dropshipping 时：

  整体优先级调整：
    • SEO 在 Dropshipping 阶段的整体优先级较低
    • 原因：SEO 见效周期 3-6 个月，Dropshipping 需要快速现金流
    • 温和提示：
      「SEO 是一项长期投资，通常需要 3-6 个月才能见效。
      当前阶段建议优先把精力放在付费广告上。
      以下 SEO 建议依然有效，你可以根据精力选择性执行。」

  提升优先级（如果用户坚持做 SEO）：
    ├── 产品页基础 SEO（最高优先级）：
    │   • Title Tag 和 Meta Description 优化
    │   • 产品描述中自然融入关键词
    │   • 图片 Alt Text 优化
    └── 技术 SEO 基础（高优先级）：
        • 网站速度优化（影响广告质量分）
        • 移动端适配（大部分流量来自移动端）
        • 基础 Schema Markup

  降低优先级（但仍然提供）：
    • 内容营销和博客矩阵（需要持续投入时间）
    • 反向链接建设（需要品牌知名度基础）
    • 长尾关键词矩阵（需要大量内容支撑）

当 supply_chain_mode = wholesale 时：
  提升优先级：
    • B2B 关键词策略（最高优先级）：“bulk order”“wholesale supplier”“B2B {product}” 等采购意图关键词
    • 批发专属落地页 SEO（高优先级）：独立的批发询价页、MOQ 说明页、合作案例页
    • 行业百科型内容（高优先级）：通过行业指南、白皮书吸引 B2B 决策者
  降低优先级（但仍然提供）：
    • 冲动购买型内容（B2B 决策更理性，不适用「限时抢购」型 SEO 策略）
    • 社交媒体驱动的 SEO（B2B 客户更依赖搜索引擎而非社交媒体）

当 supply_chain_mode = manufacturing 时：
  提升优先级：
    • 品牌故事型内容（最高优先级）：生产工艺、质量控制、工厂故事等差异化内容
    • 长尾关键词矩阵（高优先级）：制造商产品线稳定，适合投资深度内容矩阵
    • 反向链接建设（高优先级）：制造商有独特的故事角度，更容易获得媒体引用
  降低优先级（但仍然提供）：
    • 快速测品型 SEO（制造商产品线稳定，不需要频繁测试新品）
    • 低价引流策略（制造商品牌应保护价格形象）

当 supply_chain_mode = dtc 时：
  → 保持现有所有 SEO 策略和优先级不变（DTC 是系统的默认设计目标）
```

## 六、前置条件检查与 Level 0 边界

```
执行类任务最低门槛：
  → 唯一硬性门槛：需要知道「卖什么产品/什么品牌」
  → 如果用户连产品都不说，温和地告知：
    「为了给出有针对性的建议，我至少需要知道你卖什么产品。
    能简单说一下吗？哪怕一个词也行。」
  → 其他缺失信息（受众、规模、预算等）用行业通用值替代并标注

Level 0 边界（无产品/无网站/纯概念）：
  → 同样欢迎使用本模块
  → 温馨提示：「SEO通常需要有网站才能发挥最大效果。
    以下内容基于通用最佳实践，当你有了网站后，
    我可以给出更定制化的方案。」
  → 不拦截、不拒绝服务、不限制路由
  → 正常响应用户请求，基于通用最佳实践给出建议
  → 同时可以提供「SEO 基础设施清单」帮助用户起步

用户拒绝提供信息时：
  → 立即停止追问，不再收集信息
  → 基于已知信息（哪怕为零）给出最佳通用建议
  → 在输出末尾附加：
    「以上建议基于通用最佳实践。如果你能提供更多业务信息，
    我可以给出更贴合你实际情况的定制方案。」
  → 绝不因为用户不提供信息就拒绝服务或反复追问
```

## 七、危机模式止血（SEO 专项）

```
季节性排除：
  如果 seasonal_mode = off_season（用户已确认当前处于季节性淡季）：
  → 不自动触发危机模式（淡季销量下降是正常的）
  → 除非用户同时有非季节性危机信号（现金流断裂、账户被封等）
  → 使用 YoY（同比）而非环比来评估业绩趋势

当 crisis_mode ≠ none 时（即 cash_crisis 或 pr_crisis），SEO 策略应聚焦在「保护现有流量」：
  ① SEO 是危机期最重要的免费流量来源，不能丢
  ② 检查是否有技术问题导致流量下降（索引、爆错、速度）
  ③ 保护品牌词排名（危机期品牌词流量是最高转化的）
  ④ 不建议大规模内容建设（危机期需要快速见效，SEO 是长线）
  ⑤ 可以做「快速赢」：修复技术问题、优化现有高流量页面的转化率

重要补充：
  以上是「止血建议」的方向指引，不是「禁止用户做其他事」。
  如果用户在危机期坚持要做非止血类的事，尊重用户意愿，正常执行。
```
