# 启动失败复盘报告模板

> 本文档为产品启动流程的参考文件，提供启动失败后的结构化复盘方法论和报告模板。

---

## 一、复盘哲学

```text
核心信念：
├── 没有"失败的启动"，只有"产生了数据的实验"
├── 每一次失败都在缩小未知空间
├── 复盘的目的不是追责，而是提取可复用的学习
├── 最昂贵的失败是没有学到任何东西的失败
└── 最有价值的复盘是能改变下一次行动的复盘
```

## 二、失败模式识别

### 2.1 匹配的失败模式

[从以下标准模式中选择最匹配的少量模式]

| 模式 | 症状 | 是否匹配 |
|------|------|----------|
| 信息断裂 | CTR 相对可接受，但 CVR 明显弱于站内承接能力 | ✅/❌ |
| 钩子失效 | 钩子率与 CTR 同时偏弱，且落后于同批素材 | ✅/❌ |
| 加购悬崖 | 加购率明显弱于站内基线或同阶段页面表现 | ✅/❌ |
| 贴纸冲击 | 加购信号尚可，但结账完成明显掉队 | ✅/❌ |
| 信任赤字 | 新品缺少社会证明，且 CVR 明显弱于预期 | ✅/❌ |
| 受众错位 | 有曝光但漏斗各层都未形成有效承接 | ✅/❌ |
| 预算不足 | 学习和验证尚未完成，现有数据不足以定性 | ✅/❌ |
| PMF 缺失 | 多渠道、多创意测试后仍未见稳定有效信号 | ✅/❌ |

### 5.3 如果 Kill

```text
Kill 决策的前提条件（出现多项强负面信号时再考虑）：
□ 多轮 Pivot / Retry 后仍无稳定 PMF 信号
□ 总投入已逼近或超过预设止损线
□ 市场/竞品环境发生根本性变化
□ 团队资源需要重新分配到更有潜力的项目
□ 客户反馈持续负面，且产品改进空间有限

Kill 后的资产回收：
├── 用户数据 → 导入其他产品的受众库
├── 创意资产 → 归档，可能在未来项目中复用
├── 学习文档 → 归档到团队知识库
└── 邮件列表 → 如果用户同意，可用于新产品预热
```
