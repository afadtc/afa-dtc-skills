# 工作模式、KPI 体系与输出格式

> 本文件为启动规划模块的内部参考文件，用于沉淀本模块的工作模式、KPI 体系与输出模板。
> **层级声明**：本文件默认作为 **internal-only** 的内部起草模板库使用，不得整段直接粘贴给用户。
> 如需整理为用户可见交付物，必须删除内部路由标签、模块代号和系统字段，只保留自然语言、业务角色与行动建议。
> **渲染规则**：本文件中的 KPI 阈值、PMF 分档、阶段时窗、钩子数量和任何 Go/No-Go 判断，默认只可作为参考区间、保守起点或待验证计划；不得写成统一健康线、固定配方或对品牌结果的确定性承诺。

## 1. KPI 体系 (Measurement & KPIs)

### 1.1 启动效率指标

| 指标 | 定义 | 使用方式 |
|------|------|----------|
| Time to First Meaningful Order Milestone | 从广告上线到首个具解释性的订单里程碑所经历的时间 | 结合品类、预算和渠道成熟度做阶段观察 |
| CPA (Cost per Acquisition) | 获客成本 | 结合产品售价、毛利结构和回收周期解释 |
| Hook Rate | 前 3 秒留存率 | 作为素材筛选参考，不写成统一胜出线 |
| Creative Win Rate | 测试创意中达标的比例 | 结合测试口径和素材质量判断 |

### 1.2 PMF 验证指标

| 指标 | 定义 | 使用方式 |
|------|------|----------|
| PMF Score | 综合 PMF 验证得分 | 作为阶段性参考，不直接写成统一放大门槛 |

**PMF Score 计算方法**：

```text
PMF Score = 加权平均分，满分 10 分

维度 1：客户痛点匹配度（权重 25%）
  ├── 高分：多数有效样本主动提及相近核心痛点
  ├── 中高分：有明显共性痛点，但集中度一般
  ├── 中低分：痛点分散或表达不稳定
  └── 低分：缺少稳定且一致的核心痛点

维度 2：支付意愿（权重 25%）
  ├── 高分：有效样本中出现较强购买意愿或明确支付信号
  ├── 中高分：存在一定支付意愿，但价格敏感明显
  ├── 中低分：支付意愿弱或依赖强激励
  └── 低分：缺少清晰支付意愿

维度 3：产品体验满意度（权重 30%）
  ├── 高分：早期体验反馈稳定正向
  ├── 中高分：总体反馈正向，但存在明显改进项
  ├── 中低分：反馈分化较大
  └── 低分：负向反馈占主导或复购意愿不足

维度 4：竞争空间（权重 20%）
  ├── 高分：存在清晰的差异化切口与未满足需求
  ├── 中高分：有一定差异化空间，但竞争并不轻
  ├── 中低分：竞争拥挤，差异化较弱
  └── 低分：市场已高度同质化，缺少明确切口

PMF Score = (维度1×0.25) + (维度2×0.25) + (维度3×0.30) + (维度4×0.20)

决策标准：
  ├── 高分段 → 可考虑进入下一阶段，但仍需结合证据完整度复核
  ├── 中间分段 → 优先针对最弱维度补强后再评估
  └── 低分段 → 倾向继续验证、产品优化或调整方向
```

### 1.3 客户质量指标

| 指标 | 定义 | 参考解释 |
|------|------|----------|
| Repeat Purchase Rate (30d) | 30 天内复购率 | 结合品类复购周期、样本量和自有历史解释 |
| Organic Mention Rate | 自发社交提及率 | 结合社交属性、产品惊喜度和自有历史解释 |
| NPS (Early Buyers) | 早期买家净推荐值 | 结合样本量、品类特征和自有历史解释 |

### 1.4 渠道效率指标

| 指标 | 定义 | 参考解释 |
|------|------|----------|
| Blended ROAS | 混合广告回报率 | 结合毛利结构、投放阶段和回收周期解释 |
| CTR (Link Click) | 链接点击率 | 结合渠道、创意形态和受众意图解释 |
| Landing Page CVR | 落地页转化率 | 区分冷流量与暖流量，并以自有历史和页面结构解释 |
| Email Capture Rate | 邮件获取率 | 结合流量质量、优惠强度和页面设计解释 |

## 2. 工作模式

### 模式 1：完整启动规划（Full Launch Plan）

```text
触发：「帮我做一个新品启动方案」「我要上市一个新产品」
执行步骤：
  Step 1 → 收集产品信息（品类/价格/USP/目标客群/预算/现有资产）
  Step 2 → 执行三维市场验证评估（§3.1），输出 Go/No-Go 决策
  Step 3 → 设计分阶段启动作战计划（§3），含时间线和里程碑
  Step 4 → 构建 MVP 广告测试方案（§4），含假设清单和钩子矩阵
  Step 5 → 设计跨渠道协同方案（§6），含预算分配
  Step 6 → 制定预热内容日历（§3.2）
  Step 7 → 输出完整启动作战手册

输出示例片段：
  ┌─────────────────────────────────────────────────┐
  │ [Brand] 新品启动作战手册                         │
  │ 准备者：[Brand] 上市策划团队                      │
  │ 日期：[Date]                                     │
  ├─────────────────────────────────────────────────┤
  │ 一、市场验证评估                                 │
│    Go/No-Go 决策：[按证据强度填写]               │
│    核心依据：[列出当前样本与验证信号的关键依据]   │

  │                                                  │
│ 二、阶段时间线                                   │
│    阶段 0（验证）：按当前验证进度填写            │
│    阶段 1（预热）：按活动窗口倒推                │
│    阶段 2（引爆）：按活动长度设置                │
│    阶段 3（优化）：按数据回流节奏设置            │

  │                                                  │
│ 三、MVP 广告测试方案                             │
│    假设 H1：[价值主张] → 按测试范围配置钩子数    │

  │    ...                                           │

```

### 模式 2：启动诊断（Launch Diagnosis）

```text
触发：「产品上线了但没有订单」「广告花了钱但没转化」
执行步骤：
  Step 1 → 收集现有数据（广告后台截图/GA4 数据/网站 URL）
  Step 2 → 执行四层漏斗诊断（§7.1）
  Step 3 → 匹配失败模式（§7.2）
  Step 4 → 遍历相关诊断决策树（§11）
  Step 5 → 输出根因报告 + 优先行动计划

输出示例片段：
  ┌─────────────────────────────────────────────────┐
  │ 启动诊断报告                                     │
  ├─────────────────────────────────────────────────┤
  │ 诊断结论：「加购悬崖」模式                       │
  │                                                  │
  │ 症状：CTR 与当前账户基线相比尚可，但加购率明显偏低 │
  │                                                  │
  │ 根因分析：                                       │
  │ 1. 广告强调"天然成分"，落地页首屏却是品牌故事   │
  │    → 信息断裂导致用户困惑                        │
  │ 2. 价格 $68 在落地页才首次出现                   │
  │    → 贴纸冲击导致跳出                            │
  │                                                  │
  │ 行动计划（按优先级）：                           │
  │ P0：重构落地页首屏，将"天然成分"作为核心信息    │
  │ P1：在广告中预先锚定价格（"每天不到 $2.30"）    │
  │ P2：添加 Beta 用户评价作为社会证明               │
  └─────────────────────────────────────────────────┘
```

### 模式 3：PMF 评估（PMF Assessment）

```text
触发：「怎么判断我的产品有没有 PMF」「启动两周了，数据怎么看」
执行步骤：
  Step 1 → 收集启动期完整数据（达到可解释的观察周期后）
  Step 2 → 构建 PMF 仪表盘（§5.1），计算五维得分
  Step 3 → 判定 PMF 信号强度（§5.2）
  Step 4 → 输出 Scale / Pivot / Kill 决策建议
```

### 模式 4：创意测试方案（Creative Testing Plan）

```text
触发：「帮我设计广告测试方案」「怎么找到最好的广告卖点」
执行步骤：
  Step 1 → 收集产品信息和现有创意资产
  Step 2 → 提炼适合当前阶段的可测试假设（§4.2 Step 1）
  Step 3 → 为每个假设构建钩子矩阵（§4.2 Step 2）
  Step 4 → 设计广告账户测试结构（§4.2 Step 3）
  Step 5 → 定义裁决标准（§4.2 Step 4）
  Step 6 → 输出完整测试方案
```

### 模式 5：预算规划（Budget Planning）

```text
触发：「$5000 预算怎么分配」「启动期预算怎么花」
执行步骤：
  Step 1 → 确认品类、目标和可用预算
  Step 2 → 参考品类基准（§9.2）确认预算是否充足
  Step 3 → 按渠道角色分配预算（§9.1 + §6.1）
  Step 4 → 设定动态调整触发器（§6.3）
  Step 5 → 输出预算分配方案 + 调整规则
```

## 3. 输出格式规范

所有输出必须遵循以下格式约定：

```text
格式规则：
├── 1. 数据表格使用 Markdown 管道表
├── 2. 决策树使用 ASCII 树状图
├── 3. 所有数字指标标注数据来源或基准依据
├── 4. 行动计划按 P0/P1/P2 优先级排序
└── 5. 每份报告开头包含“执行摘要”（3 句话概括核心发现和建议）
```

**广告钩子脚本输出模板**（英文交付物）：

```markdown
## Hook Script: [Hypothesis] — [Hook Type]

**Platform:** [TikTok / Meta / Both]
**Format:** [Video / Image / Carousel]
**Duration:** [按平台和叙事需要确定时长]

### Opening Hook (按平台前段窗口)
[Exact script for the opening seconds — this is the most critical part]

### Problem Agitation (按叙事需要展开)
[Expand on the pain point, make the viewer feel understood]

### Solution Reveal (按叙事需要安排)
[Introduce the product as the answer]

### Social Proof (按素材与证据充足度安排)
[Testimonial, results, or authority endorsement]

### CTA (按平台节奏收口)
[Clear, urgent call to action]

### On-Screen Text Overlay
- 开场: "[Hook text]"
- 中段: "[Problem text]"
- 转折: "[Product name + key benefit]"
- 收口: "[CTA + offer]"
```

## 4. ICE 优先级排序框架

当上市计划输出多个执行方案时，使用 ICE 框架进行优先级排序：

| 维度 | 评分标准 (1-10) | 上市专属考量 |
|------|----------------|------------|
| **Impact（影响力）** | 该方案对早期订单与验证目标的贡献 | 按对当前阶段目标的贡献高低评分 |
| **数据基础（Data Basis）** | 基于数据和验证的成功把握 | 按证据强弱评分，不套用固定案例等级 |
| **Ease（易实施度）** | 实施所需的时间和预算 | 按当前团队与资源条件评分 |

**排序规则**：ICE 总分 = I × C × E / 10，按总分降序排列。上市阶段优先选择最能提升验证效率且与当前资源相匹配的方案。

## 5. 季节性产品发布框架

> **激活条件**：当 Hub 传入 `seasonal_mode = pre_season` 或用户提到"旺季新品发布""季节性发布"时，本模块自动激活。

### 5.1 季节性发布时间线

```text
季节性产品发布三阶段框架：

Phase 1: Pre-Launch（按季节窗口和供应周期提前规划）
  阶段 A：市场验证
    → 预售页面上线，收集等候名单
    → 社媒预告，测试市场反应
    → 达人/KOL 提前寄样
  
  阶段 B：内容准备
    → 拍摄产品图片和视频
    → 准备足以支持首轮测试的广告素材库
    → 编写邮件/SMS 预热序列
  
  阶段 C：渠道预热
    → 邮件列表预热（品牌故事 → 产品预告 → 倒计时）
    → 社媒内容节奏加速
    → 广告预算开始按表现和库存承接能力逐步提升
  
  阶段 D：最终准备
    → 库存确认到位
    → 网站/落地页最终检查
    → 广告 Campaign 设置和测试
    → VIP 早鸟通知发送

Phase 2: Launch（按活动长度执行）
  启动初期：爆发期
    → 全渠道同步发布
    → 邮件 + SMS + 社媒 + 广告 全力推送
    → 实时监控库存和转化数据
  
  后续阶段：优化期
    → 根据数据快速调整广告素材和出价
    → 补充表现好的 SKU 库存
    → 收集首批客户评价

Phase 3: Post-Launch（按复盘与库存消化窗口执行）
  → 复盘：哪些有效、哪些无效
  → 收集客户反馈和评价
  → 数据归档到 Brand Brain
  → 为下一个季节性发布积累经验
```

### 5.2 营销日历驱动的发布节奏

```text
Four Peaks 框架（来源：CTC）：

电商年度高峰：
  Peak 1: 年内首个关键销售窗口（按品类与市场节奏定义）
  Peak 2: 年内第二个关键销售窗口（按品类与市场节奏定义）
  Peak 3: 年内第三个关键销售窗口（按品类与市场节奏定义）
  Peak 4: 年内第四个关键销售窗口（按品类与市场节奏定义）

每个 Peak 的准备节奏：
  T-[提前窗口]: 产品和库存确认
  T-[素材窗口]: 创意和素材准备
  T-[预热窗口]: 渠道预热开始
  T-[放量窗口]: 广告预算按表现逐步提升
  T-[通知窗口]: VIP 早鸟通知
  T-0: 全渠道发布
  T+[复盘窗口]: 复盘和数据归档

Peak 之间的"谷底期"：
  → 不是什么都不做的时间
  → 是准备下一个 Peak 的时间
  → 参考 afa-ops 的淡季运营六大支柱
```

### 5.3 GTM 产品发布复盘框架

| 复盘维度 | 关键问题 | 数据来源 |
|---------|---------|---------|
| **销售表现** | 实际 vs 预测？哪些 SKU 超预期/低于预期？ | Shopify / GA4 |
| **渠道效率** | 哪个渠道贡献最多？获客效率是否符合当前预算与毛利约束？ | 广告后台 / UTM |
| **创意表现** | 哪些素材 CTR 最高？哪些 Hook 最有效？ | Meta / TikTok Ads |
| **库存管理** | 是否断货？是否有滞销？ | 库存系统 |
| **客户反馈** | 评价趋势？常见投诉？NPS？ | 评价平台 / 客服 |
| **团队执行** | 时间线是否按计划？瓶颈在哪里？ | 项目管理工具 |
